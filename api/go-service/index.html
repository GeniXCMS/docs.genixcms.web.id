    Go API Service Integration¶

    GeniXCMS 2.4.0 introduces a Hybrid Backend Architecture that allows high-performance read operations via a dedicated Go microservice. This service is designed to handle intensive GET requests while offloading load from the primary PHP engine, while still allowing PHP to "decorate" the data for a rich UI experience.

    Overview¶

    The Go API Service acts as a high-speed read-only accelerator for the GeniXCMS ecosystem. It supports:

    • Fast Execution: Compiled binary performance (query times < 10ms).
    • Hybrid Rendering: Go handles massive raw data fetching, while PHP "decorates" it with HTML badges, actions, and formatting before sending to the browser.
    • Universal Dynamic Engine (PnP): Automatically serves any database table via mapping without extra Go code.
    • Support Core Handlers:
      • Posts: Listing and single view with URL normalization.
      • Categories: Taxonomies and hierarchy.
      • Tags: High-speed autocomplete for post tags.
      • Stats: Unified dashboard analytics (counts).
      • Version: Direct system version retrieval.
      • Search: High-performance text searching across posts.
      • Updates: System, module, and theme update checks with marketplace synchronization.
      • User: Admin user listing with role and status filtering.
      • Nixomers (Orders): High-speed Order listing with SQL JOIN support.
    • Seamless Switching: Can be toggled on/off from the Admin Panel.
    • Smart Fallback: Automatically reverts to PHP if the Go service is down or returns an error.

    Configuration¶

    1. Go Service Environment (.env)¶

    You must configure the .env file inside the go-service directory:

    # Server Configuration
PORT=4544
GIN_MODE=release

# Database Configuration
DB_DRIVER=sqlite3  # alternatives: mysql, postgres
DB_HOST=127.0.0.1
DB_PORT=3306
DB_USER=root
DB_PASS=password
DB_NAME=genixcms
DB_FILE=./genixcms2.db  # For sqlite3
DB_PREFIX=              # Table prefix if any

# Authorization
GX_SECRET=your_super_secret_handshake_key

    2. Admin Panel Settings¶

    Navigate to Settings → API Service in your GeniXCMS dashboard:

    • Active Driver: Select External Go Service.
    • Service Endpoint URL: The address where your Go binary is running (e.g., http://localhost:4544).
    • Shared Authorization Secret: Must match the GX_SECRET in your .env file.
    • Auto PHP Fallback: Recommended. Reverts to standard API if Go is unreachable.

    Universal Dynamic Resource Engine (Plug n Play)¶

    GeniXCMS Go Service includes a powerful dynamic handler that automatically maps URL parameters to database tables. This enables instant high-performance read support for all third-party modules.

    Resource Mapping¶

    Requests are mapped as follows:

    • GET /ajax/my_table → SELECT * FROM my_table
    • GET /ajax/my_table/5 → SELECT * FROM my_table WHERE id = 5

    Feature: Custom Table Mapping (PnP Hook)¶

    If your AJAX resource name (e.g., inventory) is different from your database table (e.g., nix_inventory), you can map it dynamically in your PHP module without changing Go code:

    Hooks::attach('go_proxy_args', function($args) {
    if ($args['resource'] === 'inventory') {
        $args['table'] = 'nix_inventory'; // Tell Go to query this table
    }
    return $args;
});

    Feature: Generic MapScan¶

    The dynamic engine now uses Generic MapScan. This ensures Go can safely fetch data from any table structure (even with custom columns) without needing re-compilation.

    Technical Concept: Hybrid Rendering¶

    This is the key to maintaining a "Premium" look while having "Go" speed.

    1. Go Service handles the "Massive" work: Joining tables, filtering thousands of rows, and returning raw JSON.
    2. PHP Bridge identifies the response and "Decorates" it. For example, converting a status string into a colorful Bootstrap Badge or generating a Dropdown Action Menu.

    Security Restrictions (Blacklist & Whitelist)¶

    To prevent data leakage, GeniXCMS uses a dual-layer security approach:

    1. Explicit Whitelist: Administrators must list allowed database tables in the API Settings dashboard.
    2. Requests to tables NOT in the whitelist return 403 Forbidden.

    3. The Go service automatically syncs with this whitelist via the handshake.

    4. Mandatory Blacklist: The following tables are Hard-Blacklisted in the Go source code and can NEVER be accessed:

    5. users, options, permissions, configs, sessions.

    AJAX-Specific Handlers¶

    The Go service provides dedicated AJAX endpoints optimized for admin dashboard operations:

    Updates Handler¶

    • Endpoint: GET /ajax/updates
    • Purpose: Fetches and compares version information for core, installed modules, and themes against the marketplace.
    • Response: JSON with core version status, module/theme update availability, and download links.
    • Fallback: If marketplace API is unreachable, returns local version data without update information.
    • Performance: Scans filesystem for module/theme metadata once per request, cached by browser.

    User Handler¶

    • Endpoint: GET /ajax/user/list_users
    • Query Parameters:
    • offset: Pagination offset (default: 0)
    • num: Records per page (default: 10)
    • q: Search query for username/email
    • group: Filter by user group/role
    • status: Filter by user status (0=inactive, 1=active)
    • Response: JSON with paginated user list, total count, and metadata.
    • Features: SQL JOINs with user_detail table to retrieve country info; role-based filtering; search support.
    • Performance: O(n) query with indexed lookups on userid/email.

    Development Case Study: Nixomers Orders¶

    Imagine you want a high-performance order list with payment status.

    1. High Speed Query (Go)¶

    Go uses a specialized handler that performs a LEFT JOIN between nix_orders and nix_transactions to get the latest payment status instantly.

    2. Rich UI Decoration (PHP)¶

    Instead of sending raw JSON to the browser, the PHP Bridge iterates through the data and adds formatting:

    // In Ajax.class.php
if ($action === 'list_orders' && $resource === 'nixomers') {
    $raw = json_decode($response, true);
    foreach ($raw['data'] as &$row) {
        $currency = Options::v('nixomers_currency') ?: 'IDR';
        $row['total_formatted'] = $currency . " " . number_format($row['total'], 0);
        // Add a badge based on payment_status from Go
        $badge = ($row['payment_status'] === 'paid') ? 'success' : 'warning';
        $row['status_badge'] = "<span class='badge bg-$badge'>$row[payment_status]</span>";
    }
    $response = json_encode($raw);
}

    Usage inside PHP (Server-side)¶

    Modules can also fetch data from Go for server-side processing:

    $url = Options::v('go_service_url') . '/ajax/nix_products/5';
$headers = [
    'X-GX-Secret: ' . Options::v('go_service_secret'),
    'X-GX-Table: nix_products', // Force target table
    'X-GX-Proxy: PHP-Module'
];

$response = Http::get($url, [], $headers);
$data = json_decode($response, true);

    Deployment¶

    1. Build the Binary: 
      cd go-service
go build -o genix-api ./cmd/main.go
    2. Run as a Service (Recommended via PM2 or NSSM): 
      pm2 start ./genix-api --name "genix-hybrid-api"

    Routing Logic¶

    When the Go Backend is active:

    • GET Requests: Proxied to Go if the resource is in the Whitelist.
    • POST/PUT/DELETE: Always handled by PHP to ensure Data Integrity and CSRF/Session security.
    • Fallback: If Go returns a 404 or is unreachable, the system automatically runs the local PHP Ajax class.
